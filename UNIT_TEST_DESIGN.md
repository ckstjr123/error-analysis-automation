# 단위 테스트 설계안 (Unit Test Design)

본 문서는 `erroranalysis` 프로젝트의 주요 컴포넌트에 대한 단위 테스트 설계안입니다. 각 기능의 정상 동작뿐만 아니라 예외 상황(Edge Cases)에 대한 테스트 시나리오를 포함하고 있습니다.

## 1. `LlmErrorAnalyzer` (에러 분석기)
`LlmErrorAnalyzer`는 발생한 예외의 스택 트레이스를 필터링하고, 메서드 시그니처를 추출한 후, LLM(Flowise) API를 호출하여 에러 원인을 분석하는 핵심 역할을 담당합니다.

### 1.1 정상 케이스 (Happy Path)
- **스택 트레이스 필터링 및 분석 요청 성공**:
  - 지정한 스택 트레이스만 정상적으로 필터링되는지 검증.
  - `MethodSignatureParser`를 통해 시그니처가 올바르게 추출되는지 검증.
  - `FlowiseClient.analyzeError()`가 올바른 파라미터(question)와 함께 호출되고, 성공적인 응답을 반환하는지 검증.

### 1.2 예외 및 엣지 케이스 (Edge Cases)
- **필터링된 스택 트레이스가 빈 배열인 경우**:
  - 서드파티 라이브러리 내부에서만 발생하여 `ckstjr` 패키지가 없는 예외의 경우 빈 문자열로 처리되거나 빈 시그니처 리스트를 반환하여 안전하게 동작하는지 검증.
- **LLM API(Flowise) 호출 실패**:
  - `FlowiseClient` 호출 시 네트워크 타임아웃 또는 500 에러 발생 시 적절하게 예외가 전파되거나 처리되는지 검증.

## 2. `MethodSignatureParser` (메서드 시그니처 파서)
ASM 라이브러리를 이용하여 바이트코드를 런타임에 분석하고 메서드 파라미터와 반환 타입을 추출합니다.

### 2.1 정상 케이스
- **인스턴스 메서드 시그니처 추출**:
  - 일반적인 객체의 인스턴스 메서드 파라미터(this 제외)와 반환 타입이 올바르게 추출되는지 검증.
- **정적(Static) 메서드 시그니처 추출**:
  - static 메서드인 경우 this를 건너뛰지 않고 첫 번째 파라미터부터 정확히 추출하는지 검증.

### 2.2 예외 및 엣지 케이스
- **존재하지 않는 클래스 분석 시도 (IOException)**:
  - `ClassReader`가 존재하지 않는 클래스 파일을 읽으려 할 때 예외를 잡아서 빈 시그니처(`MethodSignature EMPTY`)를 반환하는지 검증.
- **라인 번호가 메서드 범위 밖인 경우**:
  - 스택 트레이스에서 주어진 라인 번호가 해당하는 클래스의 어떤 메서드에도 속하지 않을 경우 빈 시그니처를 반환하는지 검증.
- **로컬 변수 테이블이 없는 경우**:
  - 파라미터 이름을 추출하지 못할 때 `arg0`, `arg1` 등의 기본 이름으로 대체되어 반환되는지 검증.
